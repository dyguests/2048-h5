## Context

本仓库需新增可在移动浏览器游玩的 2048 H5。**提案要求**：可爱画风、方块移动时有位移动画（非瞬时跳格）、动画阶段有闪耀粒子、移动中的 tile 有拖尾。实现需在常见手机浏览器上保持流畅，且不依赖原生应用外壳。

## Goals / Non-Goals

**Goals:**

- 用清晰的前后端分层思路实现：**纯逻辑层**（网格、滑动、合并、随机新块）与 **展示层**（DOM/SVG 或 Canvas 上的位置与动画）解耦，便于测试与迭代动效。
- 通过 **CSS transform + requestAnimationFrame 或 Web Animations API**（或兼容两者之一）驱动方块从「逻辑起始格」插值到「逻辑目标格」，与一次滑动输入对应一轮动画队列。
- **粒子**：在滑动/合并动画的主要时段，以覆盖层（如绝对定位 Canvas 或少量 DOM 粒子）短时生成「闪耀」亮点，数量与时长受控以免掉帧。
- **拖尾**：移动中的 tile 使用伪元素渐变、叠加半透明 ghost 层、或 Canvas 短线段历史轨迹（最近 N 帧位置），形成可辨别的尾迹。

**Non-Goals:**

- 不强制账号、排行榜、后端存档（可后续扩展）。
- 不要求与微信/小程序同构；以标准 H5 为准。
- **不包含**：自有后端服务器、itch.io API 之外的自动发布流水线（可自行用 `butler`/CI，非本变更硬性范围）。

## Decisions

1. **视图技术：优先 DOM 网格 + transform 动画，粒子/拖尾用叠加 Canvas 或与 tile 同层的辅助层**  
   - *理由*：2048 方块数量少（最多 16 个活动格），DOM+CSS 易做圆角可爱样式与主题色；单独一层 Canvas 仅负责粒子和拖尾绘制，避免整盘用 Canvas 重写 UI。  
   - *备选*：全盘 Canvas 2D — 绘制统一但样式与字体管理略重。

2. **输入：Pointer/Touch 为主，键盘方向键可选（桌面调试）**  
   - *理由*：H5 场景以滑动为主；键盘提升开发体验。

3. **动画与逻辑时序：一次用户操作 → 先计算目标状态与移动/合并列表 → 再播放动画 → 动画结束后再生成新 tile 并解锁下一次输入**  
   - *理由*：与经典 2048 体验一致，且避免逻辑与动画不同步。

4. **粒子实现：预分配对象池 + 基于时间的 alpha/scale 插值**  
   - *理由*：减少 GC；便于限流（每帧最大粒子数）。

5. **拖尾实现：每移动 tile 维护短队列（最近 3～6 个采样点）绘制连线或渐变条带，动画结束清空**  
   - *理由*：实现成本低、性能好；避免全屏残留。

6. **面向 itch.io 的静态构建**  
   - **Vite** 设置 **`base: './'`**（或等价相对资源路径），使 `index.html` 与 `assets/` 在任意子路径（itch 嵌入 iframe、本地点开 zip）下仍能加载脚本与样式。  
   - **构建产物**：单页应用，入口为根目录 **`index.html`**，无必须的服务端 history fallback；本地用 `vite preview` 或静态服务器验证。  
   - **发布**：将 `dist/` 打 zip，在 itch 项目里选 **HTML**，上传 zip；游戏页使用「Run in browser」即在线试玩。  
   - *备选*：`base: '/'` 仅适合固定根域名根路径，与 itch 子路径/嵌入不一致，故不采用为默认。

## Risks / Trade-offs

- **[风险] 低端机同时开大量粒子卡顿** → **缓解**：粒子数量上限、动画合并为单段时合并爆发一次而非每格一次。  
- **[风险] 触摸与滚动冲突** → **缓解**：在游戏区域 `touch-action: none` 或等价处理，文档中在任务里明确实现点。  
- **[权衡]** DOM 多层叠加略增合成层内存 — 方块数量极少，可接受。
- **[风险] 绝对 URL 或错误的 `base` 导致 itch 嵌入白屏** → **缓解**：构建全相对路径并在 README/build 脚本中写明 zip 内容与 itch「This file will be launched」指向 `index.html`。

## Migration Plan

- 首次合入仅为新增静态站点或子目录；部署时整包发布即可。  
- 回滚：移除该目录与入口路由（若有），无数据库迁移。
